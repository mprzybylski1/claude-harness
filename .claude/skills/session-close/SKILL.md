# Session Close

You are closing a development session.

Your job is to: update session records, then hand off to Opus for a post-session review,
then commit everything.

## Commit discipline — code commits happen at ticket-close, not here

**By the time `/session-close` runs, all code changes must already be committed — one commit per ticket.**
Step 6 of this skill commits docs/ changes only. If code files still appear in `git status` when you reach Step 6, something went wrong — stage and commit them with a note, then investigate why the per-ticket commit was skipped.

The pattern for every ticket closed during a session:
1. Implementation done, tests pass
2. Move ticket file to `docs/tickets/closed/`, write Resolution
3. **Immediately commit:** `git add <code files> docs/tickets/ && git commit -m "feat(T###): <summary>"`
   (`git add docs/tickets/` stages both the new closed/ file **and** the deletion of the open/ file — never use `git add docs/tickets/closed/<ticket>.md` alone or the open/ deletion is silently left unstaged)
4. Only then move to the next ticket

This keeps each diff reviewable, makes `git bisect` useful, and ensures Opus can see the full change set during post-session review.

## Pre-check — Did `/implementation-review` run?

If this was a code session and `/implementation-review` was not run earlier in the
conversation, ask the user: "Want to run `/implementation-review` first so we can fix
issues inline instead of ticketing them?" If they decline, proceed normally.

## Step 0 — Determine the current session ID and refresh INDEX.md

Run all three commands and record their output:
```text
python scripts/tools/current_session.py
python scripts/tools/generate_ticket_index.py --session <N>
python scripts/tools/archive_session_log.py
```

The first command prints the current session ID (e.g. `S108`). Record the numeric part as `N`
and the full string as `CURRENT_SESSION`. You will use it everywhere below.
Do not guess, do not copy from memory.

Pass `--session <N>` to `generate_ticket_index.py` so INDEX.md is stamped with the correct
session even though sessions.md hasn't been updated yet (Step 1 runs after this step).
Example: if `current_session.py` prints `S108`, run:
    python scripts/tools/generate_ticket_index.py --session 108

Note: INDEX.md is also auto-regenerated by a PostToolUse hook whenever any ticket file is
written during the session, so it should already be current — this is just a final guarantee.

## Step 1 — Update docs/sessions.md

Update **two sections** of `docs/sessions.md`:

### Active Work section
Replace the current content with what was done this session:
- List files changed and what changed in each. Derive this list from
  `git diff <last-session-close-sha>..HEAD --name-only` plus `git diff HEAD --name-only`
  for any uncommitted changes. Do not copy from memory or prior session notes.
- Note any tickets opened or closed this session (by ID)
- List any known gaps surfaced or confirmed toward current phase goals
- Remaining open items: reference `docs/tickets/open/` — do not duplicate the list inline
- Active Work header must state: `**S[CURRENT_SESSION] — <one-line summary>**`

### Session Log section
Append one line (never edit existing lines):
```
S[N] YYYY-MM-DD: <one-line summary of what changed>
```
Where `[N]` is `CURRENT_SESSION` and `YYYY-MM-DD` is today's date.

**Do not omit this step.** The Opus review depends on accurate session notes.

## Step 1b — Update docs/system_state.md

Run the system state generator immediately after sessions.md is updated (phase status
must be current before the snapshot is taken):

```bash
python scripts/tools/update_system_state.py
```

This overwrites `docs/system_state.md` with a structured user-facing snapshot built
from sessions.md (phase & gate status), config.yaml (settings if present), and tickets/INDEX.md
(open tickets). No LLM involved — deterministic extraction only. If the script errors,
fix the cause; do not skip this step.

## Step 2 — Move closed tickets and write Resolution text

For any ticket being closed this session:
1. Tick all satisfied ACs (`- [ ]` → `- [x]`). For any item not completed, add
   `— DEFERRED to T[N]` or `— N/A: <reason>` inline. The `check_ticket_acs.py`
   pre-lint hook **blocks the move** if any bare `- [ ]` remains.
   **Scope reduction rule:** If ACs were deferred, the Resolution field must
   name the narrowed scope and link the successor ticket.
2. Move the file from `docs/tickets/open/` to `docs/tickets/closed/`.
3. Set the frontmatter field: `closed: S[CURRENT_SESSION] YYYY-MM-DD` — use the value
   from Step 0. The Stop hook enforces correct attribution and blocks session end on mismatch.
4. Write the Resolution section. The first token must be `S[CURRENT_SESSION] YYYY-MM-DD:`.

**After writing the Resolution, commit this ticket's work immediately** (see Commit discipline above). Do not wait until Step 6.

## Step 3 — Classify the session: code-touching or docs-only

```bash
python scripts/tools/classify_session.py
```

This prints `code` or `docs`. Record the output as `SESSION_TYPE`.

- `code` — at least one file matching a `code_paths` prefix in `harness.yaml` was changed,
  OR a safety-adjacent governance file (`docs/architecture_invariants.md`, or project
  safety config) was touched. Full Opus review required.
- `docs` — only docs, tickets, skills, CLAUDE.md, or tooling scripts changed.
  Static-analysis summary only; Opus agent skipped.

## Step 4 — Pre-rotate opus_notes.md

Run the rotation script before generating the Opus review context or calling Opus:

```
python scripts/tools/rotate_opus_notes.py
```

This archives the oldest `# Opus Review` section(s) from `docs/opus_notes.md` to the
appropriate decade-bucket archive file in `docs/archive/`, leaving exactly 1 section.
When Opus appends its review in Step 5, the file will contain exactly 2 sections.

The script is idempotent — safe to re-run if session-close is interrupted.

## Step 5 — Review: full Opus (code sessions) or static-only (docs sessions)

### If SESSION_TYPE = code

Generate context and run Opus:

```
python scripts/tools/prepare_opus_context.py
```

Then use the **Agent tool** with `subagent_type: "general-purpose"`, `model: "opus"`, and **`run_in_background: true`**.

Substitute the actual value of `CURRENT_SESSION` into the task before passing it:

```
You are doing a post-session review for [PROJECT].
This is session S[CURRENT_SESSION]. Append your review as:
  # Opus Review — S[CURRENT_SESSION] YYYY-MM-DD

## Start here — read these two files first, in order

1. `docs/opus_review_context.md` — contains the session diff, trimmed sessions.md,
   architecture invariants, **static analysis results**, and the **ticket template**.
   Do NOT run git diff or re-read sessions.md/architecture_invariants.md separately.
   Do NOT read docs/tickets/TEMPLATE.md — it is embedded in the context file.

2. `docs/opus_notes.md` — the single prior session review kept for context.

**Do NOT read any file not listed above.** If you find a potential issue that requires
reading source code to verify, write it with an explicit confidence qualifier
("diff suggests..." vs "confirmed:") rather than reading the file. The static analysis
section in the context file already answers the standard invariant check questions.

## Ticket system

Tickets live in docs/tickets/open/ and docs/tickets/closed/.

**For duplicate checking:** Read `docs/tickets/INDEX.md` — it has every open ticket ID,
title, phase, and age in one file. Do NOT read individual ticket files for duplication
checking. Do NOT read docs/tickets/TEMPLATE.md — use the template embedded in the
context file.

**During review, for each finding:**
- If already tracked by an open ticket (per INDEX.md): note the ticket ID. Do not duplicate.
- If a new finding worth tracking: create a ticket in docs/tickets/open/ using the template
  from the context file. Number it T[N+1] where T[N] is the highest existing ticket number
  across open/ and closed/.
- If a finding was resolved this session: move the ticket file from open/ to closed/ and fill
  in the Resolution and closed fields.

**Ticket cap: open at most 2 new tickets per session.** If you find more issues, list
the remainder as bullets in your `opus_notes.md` section under "## Minor Issues (not
ticketed)" rather than creating files. Quality over quantity.

**Severity guide:**
- critical: invariant violation, must fix before next session
- high: architectural drift or safety regression
- medium: silent failure, phase-goal blocker, or correctness issue with visible impact
- low: code quality, minor correctness issue, housekeeping

## What to look for

**Invariant violations — use the static analysis section, not source file reads:**
The context file's "Static analysis" section pre-answers the configured checks for this
project (see `harness.yaml` `static_analysis_checks`). Common checks include:
- eval()/exec() in production code (if configured)
- Append-only datastore mutation (if configured)
- Deprecated API usage in production code
- SKILL.md bash block syntax
- Test file syntax errors

For invariants NOT covered by static analysis, note them with a confidence qualifier:
- Any code that bypasses a safety check via a flag or override
- Any halt/circuit-breaker path that auto-resumes without human action
- Any place where a config change silently weakens a protection

**Architectural drift:**
- Changes that weaken fail-closed semantics, even subtly
- New parameters or config flags that allow bypassing safety checks
- Trust boundary erosion
- Test coverage gaps on safety-critical paths introduced this session

**Implementation quality:**
- Silent failures: code that logs a warning but continues when it should halt
- Off-by-one errors in thresholds
- Race conditions or unexpected state mutation in the main execution path
- Any place where an exception is caught and a default value returned instead

**Optimism in session notes:**
- Compare what the Active Work section claims was done against the actual diff
- Flag any "fixed" items that look incomplete or have deferred TODOs

## Output format

**IMPORTANT: Append your review section to `docs/opus_notes.md` and then STOP.**
Do NOT rotate, archive, delete, or restructure any existing content in `docs/opus_notes.md`.
Do NOT use `sed`, `awk`, shell commands, or Python scripts to manipulate `docs/opus_notes.md`.
The rolling-window rotation has already been done before you were called.
Append using the Read + Edit or Write tool only.

Structure:

---
# Opus Review — S[CURRENT_SESSION] YYYY-MM-DD

**Scope:** <one line describing what commits/files were reviewed>

**Files reviewed:** <list the key files>

---

## Invariant Violations
(Must be fixed before the next session starts. Reference ticket IDs.)

## Architectural Concerns
(Drift, weakened semantics, trust boundary issues. Reference ticket IDs.)

## Bugs & Implementation Issues
(Concrete code problems. Reference ticket IDs.)

## Session Notes Discrepancy
(Where Sonnet's session notes don't match the actual diff.)

## Tickets Opened This Session
(List new ticket IDs and one-line descriptions.)

## Tickets Closed This Session
(List closed ticket IDs and one-line resolution summaries.)

## Suggested Next Session Focus
(Specific, actionable. Reference ticket IDs where applicable.)

## Clean
(Anything that looked risky but held up under scrutiny.)
---

Be direct. Do not soften findings. If something violates an invariant, say so explicitly.
```

### Step 5b — Proceed with Step 6 prep while Opus runs in background

After spawning the Opus agent with `run_in_background: true`, the main thread is free.
Use this time to prepare the Step 6 commit without blocking:

1. Verify `docs/sessions.md` Active Work and Session Log entries are correct
2. Verify `docs/system_state.md` was updated by Step 1b
3. Verify all closed ticket files are in `docs/tickets/closed/` with correct `closed:` and Resolution fields
4. Verify `docs/tickets/INDEX.md` is current (generated in Step 0)

**Do NOT commit yet.** The Step 6 commit must include `docs/opus_notes.md`, which Opus is
still writing. When the background notification arrives confirming the Opus agent has finished,
check that `docs/opus_notes.md` now contains the new `# Opus Review — S[CURRENT_SESSION]`
section, then proceed to Step 6.

If the background agent fails or never notifies: read `docs/opus_notes.md` directly to check
whether the review section was written. If missing, re-run the Opus agent call (not in
background this time) before committing.

### If SESSION_TYPE = docs

Skip `prepare_opus_context.py` and the Opus agent call. Run the static analysis checks
directly (no context file needed):

```bash
python scripts/tools/run_static_analysis.py
```

This prints each check result and exits 0 (all PASS) or 1 (WARN or FAIL).
If it exits 1, escalate to a full Opus review regardless of SESSION_TYPE.

If all static analysis checks PASS, append a brief summary entry to `docs/opus_notes.md`:

```markdown

# Opus Review — S[CURRENT_SESSION] YYYY-MM-DD

**Scope:** Docs-only session — no production code changed. Static analysis only.

**Files reviewed:** `run_static_analysis.py` output (configured checks).

---

## Invariant Violations

None. Static analysis PASS on all checks. No production code changed this session.

## Architectural Concerns

None. Docs/governance/skill files only.

## Suggested Next Session Focus

[Copy the Suggested Next Session Focus from the prior Opus review in opus_notes.md —
it remains the standing priority until a code session addresses it.]
```

This entry counts as the session review. The rolling window rotation still applies —
`rotate_opus_notes.py` was already run in Step 4, so the file now contains the prior
code-session review + this docs summary = 2 sections.

**When to override and run full Opus despite SESSION_TYPE = docs:**
- Any static analysis check shows WARN or FAIL
- Session touched `CLAUDE.md`, `docs/architecture_invariants.md`, or safety-adjacent
  governance files defined in `harness.yaml` — warrant a full review even if no code changed

## Step 6 — Commit docs/ changes

All code changes should already be committed per-ticket (see Commit discipline above).
This step commits only the session-close documentation changes.

To get the correct commit message prefix, pass the N recorded in Step 0:
```bash
python scripts/tools/session_close_commit_msg.py --session N
```
This prints `docs: S[N] session close — `. Append your summary.
**Do not call this without `--session N`** — by Step 6, sessions.md already contains the
S[N] log entry, so the no-arg form would return S[N+1].

```bash
git add docs/sessions.md \
        docs/system_state.md \
        docs/opus_notes.md \
        docs/archive/ \
        docs/tickets/INDEX.md \
        docs/tickets/open/ \
        docs/tickets/closed/
git commit -m "docs: S[CURRENT_SESSION] session close — <one-line summary>"
```

If `git status` shows modified code files (outside `docs/`) at this point, stage and commit
them first with a `fix:` prefix, then run the docs commit. Note the anomaly in the session log.

The commit message must start with `docs: S[CURRENT_SESSION] session close` — this exact
prefix is what `prepare_opus_context.py` uses to locate the prior session-close boundary
in `git log`.

## What good looks like

**Hook-enforced (will block session end or tool use if violated):**
- All closed tickets have correct `closed: S[CURRENT_SESSION]` attribution — Stop hook checks this
- No bare `- [ ]` ACs in tickets moved to `closed/` — PreToolUse hook blocks the move

**Manual checks:**
- `docs/system_state.md` updated (`update_system_state.py` run in Step 1b)
- `sessions.md` Active Work header says `**S[CURRENT_SESSION] — ...**`
- `sessions.md` Session Log has a new entry for `S[CURRENT_SESSION]`
- All ticket Resolution sections start with `S[CURRENT_SESSION] YYYY-MM-DD:`
- **If phase gate status changed:** no gate threshold was lowered mid-flight after results were known. Threshold changes must predate the test run via a separate ticketed decision.
- `opus_notes.md` contains exactly 2 `# Opus Review` sections after the session
- New findings have ticket files in `docs/tickets/open/`
- Resolved items are in `docs/tickets/closed/` with resolution notes
- If Opus found Invariant Violations (code session), they are in tickets marked `severity: critical`
- Commit message starts with `docs: S[CURRENT_SESSION] session close — ...`
- For docs-only sessions: the opus_notes entry is a static-analysis summary, not a full review

## Do not use this skill when

- Nothing was changed at all — pure question-only session with zero file edits.
